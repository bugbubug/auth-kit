# Open risks — @bugbubug/auth-kit

- KV has no native atomic increment, so OtpStore.incrementAttempts is a read-modify-write `set` on TOKENS KV; under concurrent verify attempts the attempt counter can undercount (a few extra guesses past maxAttempts). The 60s resend throttle + 6-digit/10min window bound the practical exposure, but a D1-backed OtpStore (one new table -> hard-rule #2 lockstep) would be needed for strict atomic lockout if that risk is unacceptable.
- KV is eventually consistent: a code written by /start may not be immediately readable by a near-instant verify on a different colo edge. Acceptable for human email round-trips (seconds) but worth noting; an injected Clock can't fix replication lag.
- Dev/e2e email verification WITHOUT egress: with NoopEmailSender the code is generated+stored but never delivered, so e2e can't read it from an inbox. Either (a) inject a deterministic test CodeGenerator and assert a known code, or (b) keep email at 501 in default mode (gate on EMAIL_SENDER_FROM) and cover the OTP flow only in unit tests with an in-memory store. The spec chooses the 501-gate by default; flipping dev-on needs a decision on how the code is surfaced to e2e.
- Cloudflare Email Sending binding shape: `send_email` requires a verified destination_address (Email Routing) and the recipient must be a verified/allowed address in some plans — true open transactional sending to arbitrary users may require the REST API / a third-party (Resend/Postmark) instead of the binding. The EmailSender port abstracts this, but the CfEmailSender adapter + wrangler binding may need to become a REST-based adapter; confirm the CF account's Email Sending capabilities before wiring real delivery.
- Google `email` claim: id_tokens are not guaranteed to carry `email`/`email_verified` unless the GIS request includes the email scope. The verifier returns missing_email/email_unverified rather than throwing, but emo's per-product user requires an email for the projection — confirm the GIS button requests email scope so VerifiedIdentity.email is always present.
- jose as a transitive dep compiled from raw SDK TS by wrangler/esbuild: jose ships ESM and uses WebCrypto, which works in Workers, but the consumer bundler must resolve `github:` deps' own node_modules. Verify `pnpm --filter @app/api build` (wrangler dry-run) bundles jose cleanly before relying on the git-tag install.
- git-tag immutability: `github:bugbubug/auth-kit#v1.0.0` is only as frozen as the tag; if the tag is force-moved the frozen contract silently shifts. Recommend treating tags as immutable and bumping to v1.0.1 for any change, and consider committing the resolved commit SHA in pnpm-lock.yaml as the real pin.
